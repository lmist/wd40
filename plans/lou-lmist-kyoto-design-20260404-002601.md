## Problem Statement

wd40 is a markdown editor + live mindmap visualizer built on Tauri with vim bindings, 24 themes, tabs, and Rust-powered parsing. The architecture is right. The features are right. But it doesn't yet feel like it was made in Cupertino, drive like a Porsche, or provide the buttery smooth solace its creator needs. Every surface needs a premium craft pass.

## What Makes This Cool

wd40 sits at the intersection of art and science — a hacker's tool with an artist's soul. Vim power, mindmap thinking, Apple-grade craft. Richard Stallman and Steve Jobs had a baby. Jony Ive was the midwife. Nothing else in the market combines this level of keyboard-driven power with genuine visual warmth and a live spatial representation of your thinking.

This isn't a productivity tool. It's a sanctuary. Solace.

## Constraints

- Tauri v2 desktop app (macOS primary)
- Vanilla TypeScript + Vite frontend, Rust backend
- CodeMirror 6 editor, markmap-view for mindmap SVG
- Must maintain vim mode, tmux shortcuts, tab system
- No new features — only refinement of existing surfaces
- Performance is non-negotiable: 60fps animations, 0ms perceived keystroke latency
- HIG principles are foundational (Xerox PARC → Apple lineage): direct manipulation, immediate feedback, spatial consistency, objects with weight and place

## Premises

1. **The core loop is right** — markdown left, mindmap right, vim bindings, tabs. No new features. Architecture stays, craft levels up.
2. **Every pixel matters** — typography, spacing, animation timing, color temperature, transitions, scrolling feel. Subtract, don't add.
3. **Solace means warmth, not minimalism** — not iA Writer's cold white void. A warm, inhabited space. The warm color palette direction (commit ef0bf10) is the right instinct.
4. **Performance IS the design** — 0ms perceived latency on keystroke-to-mindmap. Buttery 60fps. The Rust parser was the right call. Animations cannot drop frames.
5. **The audience is the creator** — design decisions optimize for one person's taste. If it feels right, it's right.
6. **HIG swims in our genomes** — the Xerox PARC → Apple lineage of interface design is not a style guide, it's ergonomics. Direct manipulation, immediate feedback, spatial consistency, objects with weight and place.

## Approaches Considered

### Approach A: The Full Ive
Comprehensive premium pass across all surfaces at once. Typography, spacing, color, animations, mindmap rendering, transitions, scrolling, toolbar chrome. Every element audited and refined in one sweep.
- Effort: L (~2-3 hours CC)
- Risk: Low (refinement, not restructuring) but large diff is harder to review
- Pro: Coherent result, fastest to "solace"
- Con: Hard to course-correct mid-pass

### Approach B: Layer by Layer (CHOSEN)
Break the premium pass into 4 focused layers, each independently shippable:
1. Typography & spacing
2. Animation & transitions
3. Mindmap beauty
4. Chrome & details
- Effort: L (~4-5 hours total — Layer 1: ~45min, Layer 2: ~90min, Layer 3: ~90min, Layer 4: ~45min)
- Risk: Low (each layer is reviewable and revertable)
- Pro: Visible progress after each layer, can stop when it feels right
- Con: Cross-cutting concerns span layers

### Approach C: Design System First
Create DESIGN.md specification before touching any UI, then implement against the spec.
- Effort: XL (~4-5 hours)
- Risk: Medium (over-engineering for a personal tool)
- Pro: Most rigorous, prevents drift
- Con: Slow to first visible improvement

## Recommended Approach

**Approach B: Layer by Layer** — visible progress fast, course-correct between layers, start with typography because it's the thing you stare at most.

### Layer 1: Typography & Spacing
**Goal:** Every text element on screen feels intentional and balanced.

- **Editor font:** Audit current Google Fonts integration. Pick a default monospace that has personality without sacrificing readability. Candidates: JetBrains Mono (ligatures, humanist feel), Berkeley Mono (warm, distinctive), IBM Plex Mono (clean, good 0/O/o distinction). The saved font preference is respected, but the default matters.
- **Line height:** Editor line-height should breathe. Not cramped, not double-spaced. ~1.6-1.7 for the editor body.
- **Spacing rhythm:** Establish a 4px base grid. All padding, margins, gaps should be multiples of 4. Audit tab bar, statusbar, toolbar, editor padding, mindmap padding.
- **Type scale:** Statusbar text, tab names, toolbar labels, modal text — each should have a clear size/weight that communicates hierarchy without shouting.
- **Heading markers:** The heading markers (commit 58fd3e7) should feel integrated, not bolted on.

### Layer 2: Animation & Transitions
**Goal:** Everything flows. Nothing teleports. Nothing stutters.

- **Theme switching:** Transition chrome/UI colors via CSS custom properties (~200ms ease-out). Syntax highlighting swaps instantly (CodeMirror theme objects can't be CSS-transitioned). Add a brief opacity fade (150ms) on the editor container during the swap to soften the cut.
- **Panel resize:** The divider drag should feel physical. Minimum panel width of 200px with an elastic snap-back if dragged below. Content reflows smoothly during drag (no layout thrash).
- **Tab switching:** Content transition (subtle fade or slide, ~150ms).
- **Mindmap updates:** When the user types and the mindmap redraws, nodes should animate to their new positions, not pop. Use markmap's `duration` option on the Markmap instance (default 500ms, try 300ms). Custom easing is not exposed — do NOT fork markmap for easing. If 300ms duration feels acceptable, ship it. Easing customization is out of scope.
- **Toolbar/statusbar interactions:** Button hover states with subtle scale or opacity shift. Focus rings that feel native.
- **Modal (vimrc):** Fade + scale-up on open, reverse on close. Not instant.
- **Scroll:** Editor scrolling should feel weighted, like paper sliding under glass.

### Layer 3: Mindmap Beauty
**Goal:** The mindmap is art, not a diagram.

- **Branch curves:** Adjust markmap's `spacingHorizontal` and `spacingVertical` options to give curves more room. If the built-in linkShape still feels mechanical after spacing adjustments, accept it and move on. Post-render SVG path overrides would be clobbered on every re-render. File as a future markmap upstream feature request if needed.
- **Node styling:** Rounded, warm, consistent with the editor's color temperature. Text inside nodes should use the same typographic care as the editor.
- **Color palette per theme:** Audit existing `branchColors` arrays in `themes.ts` for each theme. Ensure palettes are curated rather than arbitrary. Some themes already have good palettes — focus on the ones that feel off.
- **Spacing:** Nodes shouldn't crowd. Generous whitespace between branches.
- **Expand/collapse animation:** Nodes growing/shrinking should feel like breathing.
- **Fit-to-view:** The zoom-to-fit animation should ease in/out, not snap.

### Layer 4: Chrome & Details
**Goal:** Every piece of UI chrome is invisible until needed, beautiful when noticed.

- **Tab bar:** Tabs should feel like physical objects. Active tab clearly differentiated. Close affordance subtle until hover.
- **Statusbar:** vim mode indicator, file name, pickers — visually balanced, not cramped. The statusbar should feel like a dashboard instrument cluster.
- **Toolbar buttons:** Icon-only, with tooltip. Hover/active states that feel native macOS. Consistent size and spacing.
- **Divider:** Thin, elegant, with a clear grab affordance on hover. Maybe a subtle shadow or gradient to give depth between panes.
- **Vimrc modal:** The modal should feel like a well-crafted preferences panel. Proper padding, aligned elements, a polished close button.
- **Title bar:** Traffic light padding (commit ccb03fd) is already handled. Ensure the title bar blends seamlessly with the tab bar.
- **Selection/highlight colors:** Editor selection, search highlights, active line highlight — all should use theme-aware colors that feel warm and intentional.

## Open Questions

1. Should the default theme lean warm-dark (current Tokyo Night direction) or should we pick a new signature dark theme that's uniquely wd40?
2. How much animation is too much? "Buttery" can tip into "distracting." The right answer is probably: animate structural changes (layout, expand/collapse, theme switch), don't animate content changes (typing, cursor movement).
3. Should the mindmap have a subtle background texture or gradient, or stay flat? Flat is more Apple. Subtle texture is more "warm inhabited space."
4. Is the current warm color palette (commit ef0bf10) the right base, or should we evolve it?

## Success Criteria

**Per-layer exit checks:**
- Layer 1: All spacing values are multiples of 4px. No font-size below 11px. Line-height between 1.5-1.8 on editor body.
- Layer 2: No CSS transition exceeds 300ms. No animation drops below 60fps in Safari WebKit inspector. Theme switch has no visible flash/flicker.
- Layer 3: All mindmap node text is legible at default zoom. Branch colors pass WCAG AA contrast against background.
- Layer 4: All toolbar buttons have consistent size (within 2px). Statusbar elements vertically centered.

**Subjective (the real test):**
- Open wd40 and feel calm. Feel home.
- Every interaction responds instantly — no jank, no stutter, no delay.
- Every pixel is intentional. Nothing looks like a default.
- Someone looking over your shoulder says "what is that?" and means it as a compliment.
- You reach for wd40 before you reach for anything else.
